5) PostgreSQL Note (Important for UAT/PROD)
---------------------------------------
Current app uses PostgreSQL.
For this to work in Azure:
- Configure backend `DATABASE_URL` to point to Azure Database for PostgreSQL.
- Backend can scale horizontally when PostgreSQL connection pooling and migrations are configured correctly.

For scale-out in UAT/PROD, use Azure Database for PostgreSQL Flexible Server with connection pooling.


6) Frontend Setup
-----------------
For each environment:
- Deploy a frontend instance (or named environment) with the correct VITE_API_BASE_URL.
- Ensure CORS alignment:
  - Frontend URL must be in backend ALLOWED_ORIGINS.

Example:
- DEV frontend -> DEV backend URL
- UAT frontend -> UAT backend URL
- PROD frontend -> PROD backend URL

Recommended split:
- DEV keeps Azure Static Web Apps for speed of iteration.
- UAT/PROD keep Azure Storage static website hosting and choose the edge tier by cost and maturity:
  - Azure Application Gateway WAF v2 for the current lower-cost regional setup.
  - Azure Front Door Standard when global routing, edge caching, or the broader future-state footprint is justified.
- Application Gateway does not provide the same turnkey managed HTTPS hostname that Front Door does, so plan on a custom frontend domain plus TLS certificate before cutting over to that mode.
- Because UAT can be IP-restricted at the edge WAF, run UAT promotion/frontend smoke checks from a self-hosted runner on the allowlisted office/VPN network.


7) CI/CD Promotion Flow
----------------------
Recommended flow:
1. Open PRs against `main` and use Conventional Commit PR titles.
2. Merge with squash after `pr-validate` passes.
3. Push to `main` runs `ci`, then `deploy-dev` deploys the commit SHA to DEV.
4. `release.yml` creates or updates a release PR with `release-please`.
5. Merge the release PR to create a GitHub Release and upload:
   - `release-manifest.json`
   - `frontend-dist.tgz`
6. Keep UAT and PROD parked while `ENABLE_NON_DEV_DEPLOYS=false`.
7. After non-dev is approved, use `promote-release.yml` for UAT and later PROD.
8. Promote the same release manifest digests between environments. Do not rebuild per stage.

Security recommendation:
- Use GitHub OIDC federation to Azure for deployments (avoid long-lived client secrets).
- Use one Azure app registration per GitHub environment, with subject claims:
  - repo:dcava30/Talenti_MVP:environment:dev
  - repo:dcava30/Talenti_MVP:environment:uat
  - repo:dcava30/Talenti_MVP:environment:prod


8) Validation Checklist Per Environment
---------------------------------------
After deployment:
1. Confirm the expected container apps are healthy for that environment.
2. Check backend /health endpoint.
3. In current lean dev, verify model-service and ACS calls are intentionally disabled unless the local services are running.
4. Verify auth works (register/login/token refresh).
5. Verify Blob upload URL generation works.
6. Verify Speech token issuance works.
7. Verify ACS token issuance/webhook handshake (if ACS features enabled).
8. Verify Azure OpenAI scoring path works.
9. Confirm frontend can call backend APIs end-to-end.
10. Confirm logs/metrics arrive in Log Analytics/App Insights.
11. If alerting is enabled for that environment, confirm Azure Monitor alert rules and Action Group delivery are configured.
